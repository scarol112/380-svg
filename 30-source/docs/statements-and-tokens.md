# DSL Statements and Token Support

Generated by reading `parser.py`, `ast.py`, and `lexer.py` directly.

Columns are every distinct token/parameter type the parser actually checks for.
Positional measurement columns are named after the AST field they populate.

---

## Notes

**Note 1 — `color/col` directive argument.**
The `color` directive takes a plain color name (WORD token) or quoted CSS string (QUOTED token), not a `C=…` prefixed token. It is mapped to the `C=<color>` column because it serves the same purpose (setting the active drawing color), but the syntax is `color red` or `color "rgb(…)"`, not `C=red`. The quoted CSS form (e.g. `color "rgb(100,200,50)"`) is explicitly handled via `tok.kind == "QUOTED"`.

**Note 2 — QUOTED label on non-text elements.**
`_extract_common` extracts a QUOTED token into a `label` field, but `LineElem`, `WallElem`, `DoorElem`, `WindowElem`, `ArcElem`, `ArrowElem`, and `CircleElem` have no `label` field in their AST nodes. The extracted value is silently discarded without error or warning. Marked **X** (not applicable) because the parser does not use it. Only `rect` and `textbox` actually store and render the label.

**Note 3 — `point/p` and `_extract_common` discarding.**
`_parse_point` calls `_extract_common`, which internally processes dash styles, QUOTED labels, COORD pairs, and `dim=on/off` — but `PointElem` has no fields for `dash`, `label`, `begin`, `end`, or `show_dims_override`. Those values are silently discarded. Writing `p dashed` or `p 1,2` produces no error and no effect. Marked **X** rather than **O** because the AST node cannot store or use them.

**Note 4 — `label/lb` and `dim=on/off`.**
`_parse_label` does not call `_extract_common` and does not check for `dim=on/off`. `LabelElem` has no `show_dims_override` field. Marked **X**.

**Note 5 — `lineto/lto` and COORD pairs.**
`_extract_common` is called on `tokens[2:]`, so COORD pairs present after the two positional measurements would be extracted into `begin`/`end` — but `LineToElem` has no `begin`/`end` fields. The values are silently discarded. Marked **X**.

**Note 6 — `include/inc` and filenames.**
The filename argument is read as a QUOTED token or a bare WORD. Marked in the `"text"` column as **O** (quoted form is accepted but bare WORD also works); in practice a filename is required or `ParseError` is raised — however the QUOTED form specifically is not required (a bare word suffices). A path with spaces must be quoted, since the tokenizer splits on whitespace.

**Note 7 — `direction/dir` raises `ValueError` for measurement tokens.**
`_parse_direction` calls `float(tokens[0].value)` directly. If the argument is a MEASUREMENT token (e.g. `dir 90'`), Python raises a bare `ValueError` instead of a clean `ParseError`. All other statements use `parse_measurement()` which handles feet/inches syntax. `direction` effectively requires a plain NUMBER.

**Note 8 — `textbox/tbox` font-size extraction.**
The parser counts all MEASUREMENT/NUMBER tokens and captures a NUMBER as font size only when `dim_count > 2` — i.e., the third or later such token. This skips the two required dimension arguments. The behavior is correct whether dimensions are written as MEASUREMENT tokens (`3' 2'`) or plain NUMBERs (`3 2`), but the logic is subtle and differs from other text elements which simply take the first bare NUMBER as font size.

**Note 9 — `textstyle/tstyle` with no arguments is a silent no-op.**
`textstyle` accepts zero arguments and produces a `TextStyleDirective` with both `size` and `font_family` set to `None`. The directive is applied without error but changes nothing.
